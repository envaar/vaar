# Release Process Deep Dive

This document records the maintainer workflow for shipping a Vaar release to production.
If you only need the basic guide, read [Release Process](./release.md) first.

## At a Glance

Vaar releases happen in two stages.

| Workflow            | Trigger                              | Handles                                                                             | Result                                     |
| ------------------- | ------------------------------------ | ----------------------------------------------------------------------------------- | ------------------------------------------ |
| `release-please`    | Pushes to `main` and manual dispatch | Versioning, changelog updates, release notes, tag creation, GitHub Release creation | A release pull request                     |
| `release-artifacts` | A published GitHub Release           | Validation and packaging                                                            | Uploaded archives and `vaar_checksums.txt` |

Vaar ships only the GitHub Release artifacts produced from the release tag.

## Release Controls

The source of truth for release behavior is:

- `release-please-config.json`
- `.release-please-manifest.json`
- `.github/workflows/release-please.yml`
- `.github/workflows/release-artifacts.yml`
- `.goreleaser.yaml`

Keep the repository configured with these guardrails:

- Protect `main`.
- Require pull request reviews before merging.
- Require status checks before merging.
- Require linear history.
- Allow squash merge only.
- Disallow direct pushes, force pushes, branch deletion and bypasses.

The repository also enforces a pull request title check. Titles must match Conventional Commits with an optional scope and optional `!` marker before the colon, and the accepted types include `feat`, `fix`, `hotfix`, `docs`, `test`, `refactor`, `chore`, `ci`, `build` and `perf`.
Release pull requests generated by `release-please` already satisfy that check.

Create a repository secret named `RELEASE_PLEASE_TOKEN` for `release-please`.
Use a fine-grained token or GitHub App token scoped to this repository with:

- `contents: write`
- `issues: write`
- `pull-requests: write`

`GITHUB_TOKEN` is sufficient for the `release-artifacts` workflow.

## Release-please Workflow

The `release-please` workflow runs on pushes to `main` and on manual dispatch.
It reads `release-please-config.json` and `.release-please-manifest.json`.

This workflow:

- opens or updates the release pull request
- derives the next version from merged conventional-commit PR titles
- updates `CHANGELOG.md`
- updates the manifest file with the new version state
- creates the version tag and GitHub Release when the release pull request merges

The tag format is the normal root version tag, such as `v0.1.0`.
The configuration keeps the tag free of component prefixes and includes the leading `v`.

## Release Artifacts Workflow

The `release-artifacts` workflow runs after GitHub publishes a release.
It checks out the published tag, runs `go test ./...` and invokes GoReleaser.

GoReleaser reads `.goreleaser.yaml` and produces the release archives plus `vaar_checksums.txt`.
It uploads those artifacts to the existing GitHub Release.

Expected release assets are:

- `vaar_linux_amd64.tar.gz`
- `vaar_linux_arm64.tar.gz`
- `vaar_darwin_amd64.tar.gz`
- `vaar_darwin_arm64.tar.gz`
- `vaar_windows_amd64.zip`
- `vaar_windows_arm64.zip`
- `vaar_checksums.txt`

Checksums are the integrity mechanism for the current release line.
Detached signatures and provenance attestations are not part of the release currently.

## Validate a Release Locally

Use the snapshot build when you want to exercise the packaging path without publishing anything:

```sh
make snapshot
```

Use the real release path from an exact version tag when you want to test the full release build:

```sh
git tag v0.1.0
make release
```

`make release` fails unless the current commit is tagged with an exact version such as `v0.1.0`.

If you want to mirror the release-artifacts workflow more closely, run the full test suite before the packaging step:

```sh
go test ./...
```

## Release Day Checklist

Do not declare the release complete until every item below is true:

- the release pull request merged with squash merge
- the release tag was created
- the GitHub Release exists and has release notes
- the `release-artifacts` workflow ran from the published release event
- the published release contains the expected archives and checksum manifest
- `vaar_checksums.txt` matches every uploaded archive
- an extracted binary reports the shipped version with `vaar --version`
- `README.md` still points at the correct assets and checksum file
- a smoke run still succeeds against a representative `.env` file

Example smoke run:

```bash
tmpdir=$(mktemp -d)
printf 'KEY=value\n' > "$tmpdir/.env"
(cd "$tmpdir" && /path/to/extracted/vaar lint)
```

## Recovering From a Bad Release

If a release ships with a mistake, prefer a follow-up release when that is enough to fix the problem.

If the release must be removed then:

1. Delete the GitHub Release and the matching tag.
2. Re-run the release flow from the corrected state.
3. Verify the replacement release notes, tags, archives, checksums and version output.

Avoid rewriting history unless the project maintainers have approved the same as the safest path forward.
